# Pre-0.4 Context Quality Design Spec

**Goal:** Finish the deterministic context-quality foundation before Dr. Context moves into broader 0.4 integrations such as MCP gates, safe fixes, agent probes, and AI-assisted remediation.

**Decision:** Keep one umbrella spec for the full pre-0.4 backlog, but execute it as separate release plans so each slice can ship, dogfood, and tune false positives independently.

**Release split:**

- `0.3.5`: runtime, tool, package-manager, and verification-command drift.
- `0.3.6`: agent instruction rule-quality and safety/workflow hygiene visibility.
- `0.3.7`: synthetic dogfood corpus, false-positive tracking, and before/after launch examples.
- `0.3.8` or `0.4.0`: context health scoring and badge only if prior findings are stable enough.

---

## Product principles

Dr. Context should stay boring in the best way: deterministic, local, read-only, evidence-backed, and quiet unless it has something useful to say.

Pre-0.4 work should make the scanner better at answering four user questions:

1. **Will my agent run the same tools my repo expects?** Covered by runtime/tool/package-manager drift checks.
2. **Will my agent verify changes the same way CI does?** Covered by verification-command conflict checks.
3. **Are my agent instructions usable, specific, and low-noise?** Covered by rule-quality checks.
4. **Are critical repo policies visible to the agent before it acts?** Covered by safety/workflow hygiene checks.

Anything that requires semantic judgment, provider-specific runtime introspection, network lookup, or LLM interpretation waits until 0.4+.

---

## Current shipped baseline

The following capabilities are already shipped and should not be reimplemented:

- Context surface registry for agent instruction docs.
- Repo fact extraction for package metadata, runtime files, CI commands, README commands, architecture docs, and workflow prompts.
- Context manifest and path-scoped effective context.
- Config, baseline, explicit suppression, SARIF, and workspace scan flows.
- Workflow prompt extraction and conservative workflow prompt findings.

Pre-0.4 checks should build on those existing facts wherever possible. Add extractors only when current facts cannot source the required evidence.

## 0.3.5: Drift and verification-command conflicts

### Scope

0.3.5 catches deterministic contradictions between repo configuration, CI, docs, and agent-visible instructions.

In scope:

- Node runtime drift between:
  - `.nvmrc`
  - `.node-version`
  - `package.json` `engines.node`
  - GitHub `actions/setup-node` `node-version`
  - deterministic README or agent-visible command references
- Package-manager drift between:
  - `package.json` `packageManager`
  - lockfile type
  - setup actions
  - docs or agent-visible commands such as `npm test`, `pnpm test`, or `yarn test`
- Verification-command conflicts between:
  - package scripts
  - CI commands
  - README commands
  - agent instruction commands

Out of scope:

- Fuzzy semantic command matching.
- Guessing equivalent tools from natural language.
- Non-Node runtimes unless the data is already extracted with deterministic evidence.
- Any network lookup of current tool docs.

### Findings

Candidate finding IDs:

- `node-runtime-drift`
- `package-manager-drift`
- `verification-command-conflict`

Severity policy:

- `error`: direct contradiction that is likely to break setup or verification, for example `packageManager: pnpm@...` plus agent docs saying `npm test` while CI uses `pnpm test` and no npm lockfile exists.
- `warning`: strong mismatch that may be intentional, for example `.nvmrc` and setup-node pin different major versions.
- `info`: weak visibility issue, for example docs mention a package manager command but canonical command evidence is incomplete.

False-positive rule:

- If evidence is ambiguous, prefer no finding.
- If evidence is weak but useful, use `info` with specific wording that it is visibility guidance, not a correctness claim.

### Data and boundaries

Reuse existing facts first:

- `runtimeVersions`
- `packageManagers`
- `ciCommands`
- `scripts`
- `commandMentions`
- `agentInstructionDocs`

If a new extractor is needed, it must produce normalized facts plus source spans. Checks remain pure and must not read files.

### Required tests

- Clean fixture with aligned `pnpm`, Node version, CI, README, and AGENTS commands.
- Direct package-manager conflict fixture.
- Direct Node major-version conflict fixture.
- Weak evidence no-finding fixture.
- Suppression/reporting test only if new finding flow differs from existing checks. Default should reuse existing reporting paths.

## 0.3.6: Rule-quality and safety/workflow hygiene

### Scope

0.3.6 checks whether instructions are useful and whether important repo policies are visible to the coding agent.

Rule-quality checks:

- Oversized instruction files.
- Duplicate or near-duplicate instruction blocks across supported instruction surfaces.
- Low-specificity guidance such as “use best practices,” “make it robust,” or “be careful” without concrete commands, files, or examples.
- Copied long docs embedded in agent instructions instead of references to canonical docs.
- Missing canonical examples when instructions tell the agent to follow a style, pattern, or existing implementation.

Safety/workflow hygiene checks:

- Secret hygiene policy exists in repo docs but is missing from agent-visible instructions.
- Destructive-action boundaries exist in repo docs but are missing from agent-visible instructions.
- Generated or distribution artifacts exist, but agent-visible instructions do not say how to avoid editing them.
- Planning, TDD, review, or verification policies exist in repo docs but are absent from agent-visible instructions.

Out of scope:

- LLM-based quality scoring.
- Stylistic critique without deterministic pattern evidence.
- Enforcing one universal AGENTS.md style.
- Flagging missing policies when there is no source policy in the repo.

### Findings

Candidate finding IDs:

- `oversized-instruction-file`
- `duplicate-instruction-block`
- `low-specificity-instruction`
- `copied-docs-in-instructions`
- `missing-instruction-example`
- `hidden-secret-hygiene-policy`
- `hidden-destructive-action-policy`
- `missing-generated-file-boundary`
- `hidden-workflow-policy`

Severity policy:

- Most findings start as `info`.
- Use `warning` only when repo evidence is strong and the gap is likely to cause unsafe or wasted work, for example repo security docs explicitly prohibit committing secrets but agent-visible docs omit any secret hygiene guidance.
- Avoid `error` in 0.3.6 unless a deterministic contradiction exists. This release is mostly visibility and quality, not hard correctness.

False-positive rule:

- Oversized checks need configurable or documented thresholds. Cursor’s documented 500-line guidance can inform the default for Cursor rule files, but generic AGENTS.md thresholds should be higher and reported as `info`.
- Duplicate checks should use exact or normalized block matching, not fuzzy semantic similarity.
- Low-specificity checks should require a strong phrase match and absence of concrete neighboring evidence such as commands, paths, or examples.
- Safety/workflow hygiene checks should only fire when a canonical repo policy exists outside agent-visible docs.

### Data and boundaries

Likely new helpers:

- Instruction text section splitter.
- Normalized paragraph/block fingerprinting.
- Policy keyword extractor for repo docs already read by scanner.
- Generated/dist artifact detector from file paths only.

Keep helpers deterministic and testable. Checks remain pure.

### Required tests

- Positive and negative tests for each finding family.
- Clean fixture remains clean.
- Edge tests for short generic phrases that should not fire when followed by concrete commands or examples.
- Privacy tests ensure source evidence does not print sensitive raw data beyond already-read synthetic fixture text.

## 0.3.7: Synthetic dogfood corpus and launch examples

### Scope

0.3.7 makes Dr. Context easier to trust publicly by adding synthetic repositories and examples that show what the scanner catches and what a fixed repo looks like.

In scope:

- Synthetic dogfood corpus under a docs or tests fixture location.
- False-positive tracking format for synthetic and sanitized aggregate dogfood runs.
- Before/after examples for:
  - AGENTS.md command drift
  - CI/doc mismatch
  - package-manager drift
  - workflow prompt risks
  - workspace scans
  - GitHub Action SARIF setup
- README or docs page explaining how to interpret findings and suppress accepted debt.

Out of scope:

- Raw private dogfood logs.
- Real private repo names, paths, or findings.
- Public claims that cannot be reproduced by synthetic fixtures or public CI.

### Data and boundaries

Synthetic corpus should be small but representative. Each synthetic repo should have:

- A short README explaining the intended signal.
- Expected finding IDs.
- A paired fixed version when useful.

False-positive tracking should be a Markdown or JSON fixture-friendly format, not a telemetry system.

### Required tests

- Corpus smoke test that scans each synthetic repo and asserts expected finding IDs.
- Docs examples should be executable or explicitly marked as illustrative.
- Secret/private path grep before any push.

## 0.3.8 or 0.4.0: Context health scoring and badge gate

### Scope

Context health scoring is useful only if prior findings are stable enough. It should not ship before 0.3.5 through 0.3.7 have enough dogfood confidence.

Candidate features:

- Deterministic context health summary.
- Trend-friendly JSON fields.
- Optional badge output or documented badge recipe.
- Clear explanation of score inputs.

Gate before implementation:

- The top findings from 0.3.5 and 0.3.6 have low false-positive rates in synthetic and sanitized aggregate dogfood.
- Score formula is deterministic and does not hide raw finding counts.
- Users can still treat findings as the primary output. Score is a summary, not the product.

Out of scope until 0.4+:

- Network-hosted score service.
- Telemetry-backed public benchmarking.
- AI-generated health summaries.

## Explicit 0.4+ backlog

These are important, but should not be started until the pre-0.4 foundation is stable:

- MCP context gate.
- Optional safe fixes with `--dry-run` before any write mode.
- Runtime agent view/probe simulation for Claude, Codex, Copilot, Cursor, OpenCode, and other adapters.
- AI remediation and prompt generation on top of deterministic findings.
- Context compaction, stale-assumption detector, handoff/checkpoint generation, and Ruler/Repomix integrations.
- Optional bundled JavaScript Action to avoid npm install-on-demand.

## Cross-release acceptance gates

Every pre-0.4 release must pass:

```bash
corepack pnpm test
corepack pnpm run typecheck
corepack pnpm run lint
corepack pnpm run build
corepack pnpm run pack:dry-run
npm publish --dry-run --access public
node dist/cli/main.js --version
node dist/cli/main.js check --json --root .
git diff --check origin/main...HEAD
```

Before public push, run a private path/token scan from local shell history. Do not persist private path patterns in public docs.

Every release must update:

- `README.md` when user-facing behavior changes.
- `CHANGELOG.md`.
- `ARCHITECTURE.md` when module boundaries, extractors, facts, or check categories change.
- `docs/roadmap.md`.
- `docs/release.md` when release smoke coverage changes.

## Open design decisions for implementation plans

The individual release plans must lock these down before coding:

- Exact Node version normalization rules for ranges such as `>=20`, `20.x`, and `lts/*`.
- Exact package-manager command canonicalization, especially `corepack pnpm` versus `pnpm`.
- Whether verification-command conflicts should reuse existing command-drift checks or add a separate check family.
- Instruction-size thresholds by surface type.
- Duplicate block minimum length and normalization rules.
- Which docs count as canonical policy sources for safety/workflow hygiene.
- Whether context health scoring waits for 0.4 or ships as a final 0.3.x release.
