# Pre-0.4 Gap Closure Design

## Goal

Close the known post-0.3.12 gaps before starting a larger 0.4-era bet such as write mode, MCP, agent probes, or integrations.

This slice should verify the published package, refresh stale planning pointers, add low-risk instruction-surface coverage, and turn larger ideas into explicit plans instead of hidden backlog assumptions.

## Product principles

- Keep Dr. Context deterministic, local, read-only, and evidence-backed.
- Prefer manifest/detection-only support before adding findings for new surfaces.
- Do not claim live-provider behavior unless an explicit future probe mode verifies it.
- Do not add write mode, MCP server behavior, AI remediation, or Action bundling in this slice.
- Keep published dogfood public-safe: aggregate outcomes only, no private paths, private repo names, raw local scan JSON, or detailed security findings.

## Scope

### 1. Published 0.3.12 dogfood

Run the published npm package, not the local working tree, against the sanitized dogfood set used during hardening.

Allowed output:

- repo class or synthetic label, not private repo names for sensitive repos;
- health score;
- total finding counts;
- finding IDs;
- false-positive or false-negative themes;
- resource diagnostic presence in aggregate.

Forbidden output:

- raw JSON reports;
- absolute private paths;
- bug bounty target names;
- exploitable details;
- secrets or customer/private data.

### 2. Refresh current-plan index

`docs/superpowers/README.md` still points at the dogfood hardening plan as active. After 0.3.12, it should point at the gap-closure plan and mark dogfood hardening as done.

### 3. Add detection-only instruction surfaces

Add low-risk recognition for instruction or context surfaces that appeared in current docs and tool guides after the earlier surface coverage work.

Initial targets:

- `.github/agents/*.agent.md` for GitHub Copilot custom agents.
- `AGENTS.override.md` as a Codex-style override/local layer surface.
- `CLAUDE.local.md` as a local/private Claude-style surface. It should be recognized carefully and documented as local-only or potentially uncommitted context, not a team contract.
- `.junie/guidelines.md` for JetBrains Junie.
- `JULES.md` for Jules-style agent guidance.
- `.claude/skills/**/SKILL.md` for task-level skill instructions.
- `.mcp.json` as MCP configuration context, manifest/fact only.

These should be recognized in manifests and docs. They should not automatically create new findings unless existing generic checks already apply safely through the normal instruction-file pipeline.

### 4. Update surface docs and tests

Update `docs/instruction-surface-coverage.md` so users can see what is recognized and which entries are only local/detection context.

Add tests for:

- path-to-surface classification;
- discovery/read inclusion for the new globs;
- manifest output where relevant;
- no false claim that a provider definitely loaded a surface at runtime.

### 5. Plan larger bets explicitly

Write or replace planning artifacts for:

- `drctx init` dry-run-first starter templates;
- safe fixes with dry-run patch output before any write mode;
- MCP context gate and agent probe trust boundaries;
- Ruler and Repomix docs-first integration recipes;
- child config inheritance decision;
- direct cross-agent instruction drift checks.

These are planning outputs only in this slice. Production code for those bets requires a separate implementation plan.

## Architecture

### Instruction surface registry

Extend `src/extractors/instructionSurfaces.ts` with new surface definitions. Keep each definition explicit:

- stable `id`;
- existing `tool` enum value when available, otherwise `unknown` unless the type model is intentionally extended;
- `scope` that matches current semantics;
- exact regex path matcher;
- glob used by workspace reads.

Do not introduce fuzzy path discovery.

### MCP config handling

Treat `.mcp.json` as configuration context, not as an agent instruction file that applies to source paths. If implementation needs a new fact category, add it behind manifest output only. Do not run MCP servers, parse remote URLs for reachability, or validate credentials.

### Local/private files

`CLAUDE.local.md` and similar local override files can influence agents but may be uncommitted. Dr. Context may recognize them when scanning the working tree, but docs must label them as local surfaces. Do not encourage committing personal or secret-bearing local context.

### Skills

Skill files are task-level context, not always-on project instructions. Recognize `SKILL.md` files as context inventory, but avoid treating every skill as an effective instruction for every source path unless a future design adds skill activation semantics.

## Data flow

```text
readWorkspace globs
  -> instruction surface classification
  -> manifest context inventory
  -> existing checks where safe
  -> docs and tests prove no live-provider claims
```

For dogfood:

```text
published npm package
  -> sanitized local roots
  -> aggregate table in chat/Obsidian
  -> no raw private artifacts committed
```

## Error handling and safety

- If a published dogfood scan exits `1`, classify whether it is due findings or runtime failure.
- If a scan exits `2` or crashes, stop and investigate before planning new features.
- If a new surface causes self-scan findings, review whether the finding is true before suppressing or changing checks.
- If a surface file may contain secrets or personal local instructions, do not quote its content in docs or release notes.

## Testing and verification

Required local gate after implementation:

```bash
corepack pnpm test
corepack pnpm run typecheck
corepack pnpm run lint
corepack pnpm run build
corepack pnpm run pack:dry-run
npm publish --dry-run --access public
node dist/cli/main.js check --json --root .
node dist/cli/main.js manifest --json --root .
```

Also run a private-path/token scan over changed public files before any commit that touches docs or fixtures.

## Out of scope

- `drctx init` implementation.
- `drctx fix` or any write mode.
- MCP server implementation.
- Agent probe implementation.
- AI-generated remediation.
- Bundled JavaScript Action implementation.
- Score badges, hosted trends, or telemetry-backed scoring.

## Deliverables

- Published `0.3.12` dogfood aggregate summary.
- Updated `docs/superpowers/README.md` current index.
- Updated instruction surface registry and coverage docs.
- Tests for new surface recognition.
- Planning artifacts for the larger bets listed above.
- Updated roadmap/backlog/Obsidian log.

## Recommendation

Implement this as one short gap-closure slice before selecting the next product bet. The code changes are low-risk and detection-first, while the larger items become explicit plans instead of hidden assumptions.

After this slice, the recommended next product bet should be chosen from fresh evidence:

1. `drctx init` if adoption friction is the dominant issue.
2. MCP context gate if agent integration demand is strong and the deterministic manifest is stable.
3. Safe fixes only after one dry-run-first write command has proven trustworthy.
