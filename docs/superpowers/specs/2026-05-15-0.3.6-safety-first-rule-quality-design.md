# 0.3.6 Safety-First Rule Quality Design

**Goal:** Ship a conservative 0.3.6 slice that improves agent safety/workflow hygiene and adds only low-noise deterministic rule-quality checks.

**Decision:** 0.3.6 combines the conservative and safety-first tracks. It prioritizes hidden policy visibility gaps and only includes rule-quality checks with exact, deterministic evidence. Broader heuristic rule-quality checks remain explicitly deferred before 0.4 so they are not lost.

---

## Scope

0.3.6 answers two questions:

1. **Can the coding agent see critical repo policies before acting?**
2. **Are agent instruction files obviously too large or duplicated?**

In scope for 0.3.6:

- `hidden-secret-hygiene-policy`
- `hidden-destructive-action-policy`
- `missing-generated-file-boundary`
- `hidden-workflow-policy`
- `oversized-instruction-file`
- `duplicate-instruction-block`

Deferred before 0.4:

- `low-specificity-instruction`
- `copied-docs-in-instructions`
- `missing-instruction-example`

The deferred checks stay in the pre-0.4 backlog and should be revisited after 0.3.6 dogfood data shows the policy and exact-duplicate checks are quiet enough.

---

## Product guardrails

- Prefer false negatives over noisy false positives.
- Use deterministic text/path/command evidence only.
- Do not use LLM judgments, fuzzy semantic similarity, or universal style opinions.
- Do not flag missing policies unless the repo already has that policy outside agent-visible instructions.
- Keep checks pure over `CheckContext`; new helpers may normalize text but must not read files.
- Findings must include source-backed evidence and actionable suggestions.

---

## Policy visibility checks

Policy visibility findings fire only when all of these are true:

1. A canonical repo policy source exists outside agent-visible instruction docs.
2. The canonical policy source contains deterministic keywords for the policy family.
3. Agent-visible instruction docs omit equivalent keywords or guidance.
4. The finding evidence points to the canonical source policy or artifact that proves the gap.

Canonical source policy docs for 0.3.6:

- `SECURITY.md`
- `CONTRIBUTING.md`
- `docs/SECURITY.md`
- `docs/CONTRIBUTING.md`
- `README.md`
- docs under `docs/` that are already discovered/read by the current scanner
- `.github/pull_request_template.md`
- `.github/ISSUE_TEMPLATE/*.yml|yaml|md`
- package metadata and ignored/generated artifact paths already visible to the scanner

Implementation must add only narrow read globs for missing canonical policy files. Do not add broad recursive `docs/**/*.md` discovery for 0.3.6.

Agent-visible instruction docs are the existing supported instruction surfaces, including `AGENTS.md`, `CLAUDE.md`, Cursor rules, Copilot instructions, and workflow prompt facts where applicable.

### `hidden-secret-hygiene-policy`

Fire when canonical docs explicitly mention secret hygiene but agent-visible docs do not mention secrets, credentials, tokens, `.env`, or equivalent safe-handling language.

Default severity: `info`.

Upgrade to `warning` only when canonical evidence is strong, for example `SECURITY.md` or `CONTRIBUTING.md` explicitly says not to commit secrets/tokens/`.env` files.

### `hidden-destructive-action-policy`

Fire when canonical docs explicitly restrict destructive or irreversible actions but agent-visible docs omit destructive-action boundaries.

Strong keywords include `force push`, `reset --hard`, `rm -rf`, `drop table`, `delete branch`, `production`, `destructive`, and `irreversible`.

Default severity: `info`. Use `warning` only for explicit prohibitions in canonical docs.

### `missing-generated-file-boundary`

Fire when generated/distribution artifacts are visible through deterministic metadata and agent-visible docs do not explain whether they should be edited.

Artifact evidence should come from package metadata or already-read non-ignored files, such as:

- `package.json` `files` entries such as `dist`, `build`, or generated output directories
- `package.json` `main`, `types`, or `bin` entries pointing into `dist/` or `build/`
- package scripts such as `build`, `prebuild`, or `generate-version` that name generated outputs
- generated-looking source files already read by the scanner when package scripts or comments prove they are generated

Do not recursively inventory ignored output trees solely for this check. If current scanner facts and package metadata cannot prove generated artifact boundaries, prefer no finding.

Default severity: `info`.

### `hidden-workflow-policy`

Fire when canonical docs contain explicit planning, TDD, code-review, or verification workflow policy and agent-visible docs omit that workflow guidance.

Examples of canonical workflow policy:

- “run tests before committing”
- “use TDD”
- “request review before merge”
- “update changelog/release notes”
- “run self-scan/release gate”

Default severity: `info`.

## Deterministic rule-quality checks

### `oversized-instruction-file`

Fire only on instruction files that exceed conservative size thresholds.

Default thresholds:

- Generic agent docs: more than 500 lines or more than 30 KB of text.
- Cursor scoped rule files: more than 500 lines.
- Workflow prompt values: more than 12 KB of prompt text.

Severity: `info`.

Suggestion: split into smaller scoped files or link to canonical docs instead of embedding long content.

### `duplicate-instruction-block`

Fire only for exact or normalized-exact duplicate blocks across supported instruction surfaces.

Normalization rules:

- Trim leading/trailing whitespace.
- Collapse internal whitespace runs to one space.
- Ignore markdown heading marker depth differences for otherwise identical lines.
- Ignore frontmatter blocks.
- Ignore boilerplate-only blocks shorter than the minimum threshold.

Minimum duplicate block threshold:

- At least 5 non-empty lines, or
- at least 300 normalized characters.

Severity: `info`.

Suggestion: keep the rule in one canonical place and reference it from narrower surfaces.

No fuzzy similarity or semantic deduplication in 0.3.6.

## Deferred checks before 0.4

The following checks remain important but should not ship in 0.3.6 because their false-positive risk is higher without dogfood calibration.

### `low-specificity-instruction`

Future design direction:

- Match a small allowlisted phrase set such as “use best practices”, “make it robust”, “be careful”, and “write clean code”.
- Suppress when nearby text includes concrete commands, paths, examples, or explicit decision rules.
- Start as `info` only.

### `copied-docs-in-instructions`

Future design direction:

- Detect very long copied sections that also appear in canonical docs or README-like files.
- Prefer exact normalized block overlap, not semantic similarity.
- Suggest linking to canonical docs instead of duplicating them.

### `missing-instruction-example`

Future design direction:

- Fire when agent docs say to follow an existing style/pattern but do not name a concrete example file or command.
- Suppress when any nearby path, glob, symbol, or example file is present.
- Start as `info` only.

These deferred checks should be planned into either 0.3.7 dogfood/examples or 0.3.8 context-health readiness, depending on dogfood noise.

## Testing strategy

Use TDD for every finding family.

Required fixtures:

- Clean repo where policy docs and agent-visible instructions are aligned.
- Secret policy hidden from agent instructions.
- Destructive-action policy hidden from agent instructions.
- Generated artifact boundary missing from agent instructions.
- Workflow/TDD/review/verification policy hidden from agent instructions.
- Oversized instruction file fixture.
- Duplicate instruction block fixture.
- Negative fixtures where generic or weak evidence does not fire.
- Negative fixtures where agent-visible docs already mention the relevant policy.
- Negative fixture where ignored `dist/` or `build/` output exists but package metadata does not prove a generated boundary.
- Duplicate-block negative fixtures for frontmatter-only overlap and short boilerplate.

Regression expectations:

- Dr. Context self-scan must remain clean.
- Suppression, JSON, text, SARIF, and workspace reporting should work through existing finding/reporting paths without custom report logic.
- If any self-scan finding appears, either fix the repo instruction visibility or document the accepted finding before release.

## Implementation notes

- Prefer extending existing text extraction facts instead of adding broad file discovery.
- If policy docs are not currently read, add the narrowest read globs needed for the canonical docs listed above.
- New helpers should live near checks or in a focused text-normalization module.
- Do not add config knobs in 0.3.6 unless a threshold proves impossible to keep quiet. Defaults can be documented first.
- Duplicate block detection should use normalized block fingerprints in a map. Do not do pairwise fuzzy comparisons.
