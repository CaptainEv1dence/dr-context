# 0.3.2 Scoped Effective Context Design

Date: 2026-05-14
Status: approved for implementation planning
Project: Dr. Context

## Purpose

Dr. Context 0.3.2 should model which agent instructions apply to a target path or workspace child. The feature turns the 0.3.1 instruction surface registry into a scoped context resolver without adding LLM calls, network calls, write mode, or vendor runtime introspection.

The core question is:

```text
Which instructions will an AI coding agent likely read for this file or candidate repo?
```

## Product goals

- Add path-scoped manifest output with `drctx manifest --path <relative-path>`.
- Add explicit workspace inheritance with `drctx check --workspace --inherit-parent-instructions`.
- Include Cursor nested rules in effective context when they apply to a target path.
- Add conservative scoped-rule findings for invalid or ineffective scoped rules.
- Preserve deterministic, local, read-only, evidence-backed behavior.

## Non-goals

- No default parent inheritance. Users must opt in with `--inherit-parent-instructions`.
- No exact vendor runtime introspection.
- No fuzzy or LLM interpretation of rule intent.
- No network calls.
- No write mode or automatic fixes.
- No broad rule-quality checks beyond scoped rule validity. Oversized, duplicate, and low-specificity instruction checks stay in later slices.
- No MCP server in this release.

## Current baseline

0.3.1 already provides:

- `AgentInstructionDocFact` with `tool`, `scope`, optional `appliesTo`, and optional `metadata`.
- A central instruction surface registry.
- Manifest instruction file metadata.
- Workspace discovery and workspace scan output.
- CI command classification and richer repo facts.

0.3.2 should build on those facts, not replace them.

## Feature design

### 1. Effective context resolver

Add a pure resolver that converts known repo facts plus a target path into effective instruction files:

```text
RepoFacts + EffectiveContextOptions -> EffectiveContext
```

Suggested types:

```ts
export type EffectiveContextOptions = {
  targetPath?: string;
  includeInherited?: boolean;
  parentInstructionFiles?: AgentInstructionDocFact[];
};

export type EffectiveInstructionFile = {
  path: string;
  type: AgentInstructionTool;
  scope: AgentInstructionScope;
  inherited: boolean;
  appliesBecause: string;
  source: SourceSpan;
};

export type EffectiveContext = {
  targetPath?: string;
  instructionFiles: EffectiveInstructionFile[];
};
```

Rules:

- Repo-root instruction files apply to all target paths inside that repo.
- Nested `AGENTS.md` applies when it is at or above the target path directory.
- The nearest nested instruction files are included, and parent nested instruction files may also be included when they are in the target path ancestry.
- Cursor nested rules under `.cursor/rules/` apply deterministically by path metadata when available, otherwise by conservative directory conventions.
- Inherited parent workspace instructions are only included when `includeInherited` is true and must be marked `inherited: true`.
- Every effective instruction has a human-readable `appliesBecause` string.

The resolver must not read files. It consumes facts already extracted by `runScan` or workspace scanning.

### 2. Path-scoped manifest

Add:

```bash
drctx manifest --path backend/src/api.ts
```

Behavior:

- `--path` is relative to `--root`.
- Absolute target paths are rejected unless they resolve inside `--root`; the manifest should then normalize them to a relative path.
- Target paths escaping `--root` are runtime/config errors with exit code 2.
- The JSON manifest keeps existing fields and adds:
  - `targetPath`
  - `effectiveInstructionFiles`

Example JSON shape:

```json
{
  "schemaVersion": "drctx.manifest.v1",
  "targetPath": "backend/src/api.ts",
  "effectiveInstructionFiles": [
    {
      "path": "AGENTS.md",
      "type": "agents",
      "scope": "repo",
      "inherited": false,
      "appliesBecause": "repo-level instruction applies to all paths under the requested root",
      "source": { "file": "AGENTS.md", "line": 1, "text": "# AGENTS.md" }
    }
  ]
}
```

Text manifest should add a compact section when `--path` is present:

```text
Effective instruction files for backend/src/api.ts:
- AGENTS.md (agents, repo) — repo-level instruction applies to all paths under the requested root
- backend/AGENTS.md (agents, nested) — target path is under backend/
```

### 3. Workspace parent inheritance

Add:

```bash
drctx check --workspace --inherit-parent-instructions
```

Default behavior stays unchanged. Workspace child scans do not inherit parent instructions unless the flag is set.

When enabled:

- Parent instruction files from the requested workspace root are passed into child scans as inherited instruction facts.
- Inherited facts must be marked in reports and manifests where exposed.
- JSON output must continue to redact workspace roots as it does today.
- Parent instruction evidence must remain source-backed and relative to the workspace root or redacted candidate root.
- Findings from inherited instructions should be conservative. Prefer false negatives if inheritance creates ambiguous ownership.

This flag is meant for monorepos where a root `AGENTS.md` defines global policy and child packages have local context.

### 4. Cursor nested rule inheritance

Support `.cursor/rules/**/*.mdc` and `.cursor/rules/**/*.md` in effective context.

Deterministic matching order:

1. If frontmatter has path/glob fields supported by the current extractor, use those fields.
2. If no usable path metadata exists, treat the rule as nested but apply conservatively based on a directory convention only when the rule path clearly names a subtree. For example, `.cursor/rules/backend/api.mdc` may apply to `backend/**` if that convention is encoded and tested.
3. If no deterministic scope can be inferred, include the rule in the general instruction surface list but not in `effectiveInstructionFiles` for a target path.

Do not guess from natural language descriptions.

### 5. Invalid scoped rule findings

Add a pure check for scoped instruction surfaces.

Initial finding IDs:

- `invalid-scoped-rule-glob`
- `scoped-rule-matches-no-files`
- `scoped-rule-too-broad`

Severity:

- Invalid glob syntax: `warning`.
- Glob matches no files: `info` by default, `warning` only when the source explicitly claims the rule should apply to a known directory.
- Too broad: `info` in 0.3.2.

Rules:

- Findings must include source evidence from the scoped rule file.
- The check must not scan outside `--root`.
- The check should use already discovered files where possible. If more file discovery is needed, it belongs in extraction/readWorkspace, not inside the check.
- No error-level findings in 0.3.2.

## CLI and report behavior

### Existing commands remain compatible

These commands must keep their current output shape unless new optional fields are added:

```bash
drctx check --root .
drctx check --workspace --root .
drctx manifest --root .
drctx discover --root .
```

### New options

```bash
drctx manifest --path <relative-or-root-contained-path>
drctx check --workspace --inherit-parent-instructions
```

If `--inherit-parent-instructions` is used without `--workspace`, the CLI should return a clear usage error with exit code 2.

## Data flow

```text
readWorkspace(root)
  -> extract repo facts
  -> resolveEffectiveContext(facts, { targetPath })
  -> buildManifest(... effectiveContext)
  -> manifestReporter

discoverCandidates(root)
  -> for each candidate: runScan(candidateRoot, inherited parent facts if flag enabled)
  -> workspaceReporter
```

The resolver and checks should stay pure. CLI parsing and file-system path validation stay at adapter boundaries.

## Testing strategy

Add fixtures for:

- Root `AGENTS.md` plus nested `backend/AGENTS.md` with `manifest --path backend/src/api.ts`.
- Workspace parent `AGENTS.md` plus child package `AGENTS.md` with inheritance disabled and enabled.
- Cursor nested rules with deterministic path metadata.
- Invalid scoped glob syntax.
- Scoped rule matching no files.
- Scoped rule matching too broadly.
- `--path` escaping the root.

Required test categories:

- Resolver unit tests.
- Manifest builder/reporter tests.
- CLI integration tests.
- Workspace scan tests.
- Check tests for scoped rule findings.
- Regression tests showing default workspace behavior does not inherit parent instructions.

## Subagent execution plan

Implementation should be split across subagents:

1. Resolver subagent: types, `effectiveContext.ts`, resolver tests.
2. Manifest/CLI subagent: `manifest --path`, JSON/text output, CLI tests.
3. Workspace inheritance subagent: `--inherit-parent-instructions`, workspace scan/report tests.
4. Scoped rules subagent: glob parsing, invalid scoped rule check, tests.
5. Review subagents: spec compliance review, code-quality review, false-positive/noise review.

Each implementation subagent should use TDD and return changed files, test commands, and known risks. The coordinating agent verifies all changes locally before release prep.

## Release criteria

- No new default findings on Dr. Context self-scan unless the finding is intentional and documented.
- Default workspace scan output remains compatible.
- `manifest --path` output is deterministic and redacts absolute roots.
- Parent inheritance is opt-in and visibly marked.
- Scoped-rule checks stay low-noise and evidence-backed.
- Published package smoke covers `manifest --path` and workspace inheritance.
