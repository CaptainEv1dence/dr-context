# 0.3.10 DX Reliability Bundle Design

## Goal

Turn the 0.3.9 launch polish into a maintainable product surface: docs stay source-accurate, first-run output teaches users what happened, the GitHub Action reliability path is explicit, and larger pre-0.4 feature bets are decomposed before implementation.

## Current state

Dr. Context 0.3.9 shipped as a launch/adoption release. The package is published, CI and release workflows are green, README onboarding is stronger, and the project now has reference docs for findings, GitHub Actions, instruction surfaces, demos, triage, and launch copy.

Two release-follow-up facts matter:

- `CHANGELOG.md` still says `0.3.9 - Unreleased`, and `docs/roadmap.md` still lists 0.3.9 launch adoption work under `Next`. This is harmless but visible post-release drift.
- The 0.3.9 implementation exposed a real maintenance risk: source-derived docs are currently protected by manual review, not tests. The finding reference should describe emitted finding IDs, not internal check-family IDs such as `coverage-signals` or `scoped-rules`.

## Tracks

### Track A: 0.3.10 DX Reliability and docs validation

This is the recommended next release. It should ship first.

It does three things:

1. clean post-0.3.9 release drift in roadmap and changelog;
2. add tests that keep docs/reference content source-accurate;
3. add `drctx explain <finding-id>` backed by a static local finding reference catalog.

Architecture:

- Add a small catalog module, likely `src/core/findingReference.ts`, containing emitted finding IDs and human-facing explanation metadata.
- Keep check execution unchanged. Checks remain pure and keep returning `Finding[]`.
- Add tests that compare the catalog against emitted finding IDs from source and docs pages.
- Add a thin CLI command that renders a single catalog entry as text or JSON.

Non-goals:

- No external docs lookup.
- No LLM-generated explanations.
- No scanner behavior changes.
- No exit-code changes.

### Track B: GitHub Action reliability

This is a separate reliability track. It should not be mixed into Track A unless it stays tiny.

The current Action is a composite wrapper that runs:

```text
npm exec --package dr-context@$DR_CONTEXT_VERSION -- dr-context check --sarif
```

That is honest and working, but it depends on npm registry availability during CI runs. The track should evaluate a bundled JavaScript Action only behind a hard gate.

Architecture:

- First extract Action argument mapping and SARIF validation expectations into tests or documented fixtures.
- Preserve inputs: `root`, `strict`, `upload-sarif`, `version`.
- Preserve SARIF output, annotations, summary, and exit code behavior.
- If bundling needs a new bundler dependency, large package-contents changes, or uncertain cross-platform path handling, stop and defer with evidence.

Non-goals:

- No scanner rewrite.
- No Action input rename.
- No behavior difference between npm-package execution and bundled execution.

### Track C: first-run product polish

This track improves what users see on first run without changing machine-readable contracts.

Architecture:

- Text reporters may add concise guidance after normal output.
- JSON and SARIF should remain stable unless a new additive field is explicitly planned.
- Coverage findings such as `no-scannable-context` and `no-agent-instructions` should get the clearest text path: what was scanned, what was missing, and what command to run next.
- The happy path may show a small scanned-context summary, but should stay quiet enough for CI logs.

Non-goals:

- No health score gates.
- No network docs links in scan output unless they are short static docs paths or README references.
- No noisy marketing copy in CLI output.

### Track D: pre-0.4 feature bets decomposition

This track plans larger features, not implementation in the same release.

Candidates:

- `drctx init` with dry-run and no-overwrite behavior.
- Optional safe fixes with `--dry-run` before any write mode.
- MCP context gate.
- Optional agent view/probe simulation.
- Ruler/Repomix integrations.

Architecture:

- Each feature gets a separate spec and implementation plan before code.
- Write modes require explicit dry-run-first design, file backup/preview policy, and tests that prevent overwriting user files.
- MCP/probe features must stay clearly labeled as optional layers above deterministic scan results.

Non-goals:

- No bundling all pre-0.4 bets into one PR.
- No write-mode implementation without a dedicated safety spec.
- No provider-runtime claims without live evidence and clear labeling.

## Recommended sequence

1. Ship Track A as `0.3.10`.
2. Execute Track C as `0.3.11` if Track A stays clean, or fold the smallest first-run text improvements into `0.3.10` only if tests remain small.
3. Run Track B as a separate reliability spike, then implement only if the gate passes.
4. Use Track D to select one pre-0.4 bet, preferably `drctx init` or `drctx explain` follow-up workflows, not MCP first.

## Success criteria

- Roadmap and changelog reflect that 0.3.9 shipped.
- CI fails if finding reference docs drift from emitted finding IDs.
- CI fails if instruction-surface docs drift from `src/extractors/instructionSurfaces.ts`.
- `drctx explain <finding-id>` works locally for every emitted finding ID.
- First-run text changes do not alter JSON/SARIF schemas or exit codes.
- Action reliability work is either implemented with parity tests or deferred with clear evidence.
- Pre-0.4 features are decomposed into separate specs before implementation.

## Self-review

- Placeholder scan: no unresolved placeholders or TODO-style requirements remain.
- Scope check: the bundle is intentionally split into four tracks so implementation can happen independently.
- Type consistency: current code names are used where known, and new names are reserved for plans rather than silently introduced in production code.
