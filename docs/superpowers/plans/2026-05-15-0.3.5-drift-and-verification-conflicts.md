## Candidate finding IDs

- `node-runtime-drift`
- `package-manager-drift`
- `verification-command-conflict`

Severity policy:

- `error`: direct contradiction likely to break setup or verification.
- `warning`: strong mismatch that may be intentional but should be visible.
- `info`: weak visibility guidance where evidence is useful but not decisive.

False-positive policy:

- Prefer no finding when evidence is ambiguous.
- Do not infer tool equivalence from prose.
- Treat `corepack pnpm` and `pnpm` as the same package-manager intent only after the normalization rule is tested.

## Locked 0.3.5 normalization decisions

### Node runtime normalization

Normalize these static Node version forms:

- `20`, `v20`, `20.11.1`, and `v20.11.1` => major `20`, confidence `high`.
- `20.x` and `20.*` => major `20`, confidence `high`.
- `>=20` => minimum major `20`, confidence `medium`.

Do not create drift findings for dynamic or unsupported values such as `lts/*`, `node`, `latest`, matrix expressions, environment variables, GitHub expressions, or unresolved YAML aliases.

A `node-runtime-drift` finding may only fire when deterministic normalized evidence cannot overlap, for example exact/static major `18` versus exact/static major `20`, or exact/static major `18` versus minimum major `>=20`.

If any side is unsupported or dynamic, emit no finding.

### Package-manager command normalization

Normalize package-manager command intent before comparison:

- `pnpm ...` => `pnpm`
- `corepack pnpm ...` => `pnpm`
- `corepack pnpm@<version> ...` => `pnpm`
- `npm ...` => `npm`
- `yarn ...` => `yarn`
- `bun ...` => `bun`

Do not infer equivalence from prose. Do not normalize `corepack enable` as a package-manager command by itself. Do not compare commands behind shell variables, aliases, Make targets, or scripts unless an existing deterministic classifier already resolves them.

A package-manager drift finding requires source-backed canonical repo intent, preferably `package.json packageManager` or exactly one JS lockfile, plus conflicting deterministic command evidence.

### Verification-command conflict rules

`verification-command-conflict` compares deterministic verification command facts only.

A finding may fire when agent-visible instructions tell the agent to run a different verification command than CI/package scripts for the same repo verification intent, for example `npm test` in AGENTS.md while `packageManager` and CI both use `pnpm test`.

Do not flag README-only weak evidence by default. README evidence can support another stronger source, but it must not create an error by itself.

Ignore shell plumbing commands such as `tee`, `cat`, `echo`, `grep`, `jq`, output redirection, `set -e`, `&&`, and line-continuation structure unless the existing command classifier already identifies a deterministic verification command.

Use existing finding/reporting/suppression/SARIF flow. Do not add custom reporting, custom suppression, or reporter-side inference.

## Guardrails

- Prefer false negatives over noisy false positives.
- Every finding needs source-backed evidence and a concrete suggestion.
- Do not print secrets or private paths.
- Do not add new default findings to Dr. Context self-scan unless explicitly accepted and documented.
- Keep checks pure; extractors own parsing and source mapping.
- Checks must receive normalized/source-backed facts or call shared pure normalization helpers. Do not duplicate ad hoc parsing inside each check.
- README-only evidence must not produce `error` findings.
- SARIF, JSON, text, baseline, and suppression behavior must come from the normal finding pipeline.
