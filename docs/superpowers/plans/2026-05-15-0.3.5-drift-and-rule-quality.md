# 0.3.5 Drift and Rule Quality Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the roadmap items deliberately deferred from 0.3.4: deterministic runtime/tool drift checks, broader verification command conflict checks, and low-noise agent rule-quality/safety visibility checks.

**Architecture:** Build on existing extracted facts from package metadata, runtime files, CI commands, README commands, and agent instruction docs. Add pure checks only; do not add network calls, LLM calls, write mode, or provider runtime introspection.

**Tech Stack:** TypeScript, Vitest, existing scanner pipeline, existing text/JSON/SARIF reporters.

---

## Why this is 0.3.5, not 0.3.4

0.3.4 should stay focused on workflow-embedded Claude prompts. The previous roadmap also paired workflow prompts with drift and rule-quality work, but bundling all of that into one release would increase noise and review cost. 0.3.5 is the next slice for the remaining deterministic checks.

---

## In scope

- Runtime version drift checks:
  - `.nvmrc`
  - `.node-version`
  - `package.json` `engines.node`
  - GitHub `actions/setup-node` `node-version`
  - README / agent-visible command references when deterministic
- Package manager drift checks:
  - `package.json` `packageManager`
  - lockfile type
  - setup actions and docs commands
- Verification command conflict checks across instruction surfaces:
  - agent docs say to run one command while package/CI expose another canonical verification command
  - keep `error` only for deterministic contradictions; prefer `warning` or `info`
- Rule-quality checks:
  - oversized instruction files
  - duplicated instruction blocks across surfaces
  - low-specificity guidance such as `use best practices` or `make it robust`
  - copied long docs instead of references
  - missing canonical examples when instructions say to follow a style or pattern
- Safety/workflow hygiene visibility checks:
  - secret hygiene instruction gaps when repo policy exists
  - destructive-action boundary gaps when repo policy exists
  - generated/dist do-not-edit guidance gaps when generated files are present
  - planning/TDD/review/verification visibility gaps when repo docs already require those workflows

---

## Out of scope

- Semantic LLM comparison of instructions.
- Non-deterministic style critique.
- Network lookup of tool docs.
- Automatic fixes.
- MCP context gate.
- Runtime agent probing.
- More workflow prompt provider support. That remains separate from 0.3.5 unless 0.3.4 leaves a tiny follow-up.

## Candidate tasks

### Task 1: Runtime and package-manager drift design

- Define exact normalized version comparison rules.
- Decide which evidence combinations produce `error`, `warning`, or `info`.
- Add fixtures for clean, warning, and no-finding cases.

### Task 2: Runtime version drift checks

- Add pure checks over existing `runtimeVersions` facts.
- Add tests with `.nvmrc`, `.node-version`, `package.json engines`, and workflow setup-node versions.

### Task 3: Package-manager drift checks

- Add checks comparing package manager facts, lockfiles, setup actions, and agent-visible commands.
- Preserve existing package-manager mismatch behavior and avoid duplicate findings.

### Task 4: Verification command conflict checks

- Extend existing command drift logic without making command matching fuzzy.
- Add tests for deterministic contradictions and weak evidence no-finding cases.

### Task 5: Rule-quality checks

- Start with low-noise `info` findings.
- Add tests for oversized, duplicate, low-specificity, copied-doc, and missing-example patterns.
- Ensure clean fixture remains clean.

### Task 6: Safety/workflow hygiene visibility checks

- Only flag when repo evidence shows the policy exists outside agent-visible docs.
- Add source-backed evidence from both policy docs and missing/weak agent surfaces.

### Task 7: Docs and release prep

- Update README support matrix and examples.
- Update CHANGELOG, roadmap, architecture docs, release smoke checks.
- Run full release verification before shipping.

---

## Guardrails

- Prefer false negatives over noisy false positives.
- Every finding needs source-backed evidence and a concrete suggestion.
- Do not print secrets or private paths.
- Do not add new default findings to Dr. Context self-scan unless explicitly accepted and documented.
- Keep checks pure; extractors own parsing and source mapping.
