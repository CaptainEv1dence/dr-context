# 0.3.9 Launch Adoption Polish Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make Dr. Context easy to understand, trust, and install in CI after the 0.3.8 health-summary release.

**Architecture:** This is an adoption release first. It changes docs, examples, and packaging reliability only; scanner semantics remain deterministic, local, read-only, and no-LLM/no-network at analysis runtime. Any GitHub Action bundling must preserve existing inputs and report flows, and any heuristic work is deferred unless it is docs-only or explicitly experimental `info` behavior.

**Tech Stack:** Markdown docs, TypeScript/Vitest if Action packaging is implemented, GitHub Actions metadata, existing Dr. Context CLI.

---

## Product decision

0.3.9 should not be “more checks by default.” The product has enough deterministic signal after 0.3.8. The bottleneck is that a new user must understand the value in under a minute and trust that CI integration will not be noisy or flaky.

Ship 0.3.9 as:

1. launch-grade README and demo docs;
2. first-run triage and finding-reference docs;
3. recognized instruction-surface coverage map;
4. GitHub Action reliability review, with a bundled JavaScript Action only if the implementation stays narrow;
5. explicit deferral of noisy heuristics and write-mode features.

---

## In scope

- README top-of-page rewrite around the concrete user promise: “Found N ways your repo lies to AI agents.”
- `docs/demo.md`: a compact, synthetic, copy-pasteable demo built from the existing dogfood corpus and before/after examples.
- `docs/triage-findings.md`: how to decide what to fix first, how not to chase health score, how to use baseline/suppression without hiding debt.
- `docs/github-action.md`: minimal and SARIF upload Action setup, permissions, local/read-only constraints, troubleshooting.
- `docs/finding-reference.md`: static local explanation of shipped finding IDs, severity, confidence, evidence, and suggested fixes.
- `docs/instruction-surface-coverage.md`: recognized local instruction surfaces by tool family and how to inspect them via `drctx manifest`.
- Optional: bundled JavaScript GitHub Action packaging if it preserves current semantics and does not turn into a platform rewrite.
- Roadmap updates that keep deferred user-desired items visible.

---

## Out of scope

- Default `low-specificity-instruction`, `copied-docs-in-instructions`, or `missing-instruction-example` findings.
- Score badges, persisted health trends, score gates, hosted score services, or telemetry comparisons.
- MCP context gate.
- `drctx init` or any write-mode template generation.
- `drctx fix`, patch generation, or AI remediation.
- Live agent probes or claims about what a provider actually loaded at runtime.
- External-doc or internet comparison during scan.

---

## Findings from roadmap review

- Launch polish is the highest-leverage 0.3.9 work. Existing docs are accurate but too reference-like for a first-time user.
- Bundled JS Action is a concrete reliability win because the current composite Action runs `npm exec --package dr-context@$DR_CONTEXT_VERSION` at workflow runtime.
- Deferred heuristics are still valuable but must not become default noisy warnings before stronger dogfood calibration.
- `missing-instruction-example` is the safest future heuristic; `low-specificity-instruction` is higher noise; `copied-docs-in-instructions` should be renamed or narrowed to deterministic repo-local duplication before implementation.
- Users also want a coverage map: “which instruction files does Dr. Context recognize for each agent family?” This should be framed as recognized local surfaces, not live runtime visibility.
- Future user-desired items remain: `drctx init`, `drctx explain <finding-id>`, safe fixes with `--dry-run`, MCP, probes, AI remediation, Ruler/Repomix integrations, handoff/checkpoint tools.

# Triage Dr. Context findings

Findings are the source of truth. Health is only a summary.

## Fix order

1. Fix `error` findings with `high` confidence first.
2. Fix unsafe or hidden policy findings before normal polish.
3. Fix command and runtime drift before wording cleanup.
4. Review `warning` findings with the owner of the repo workflow.
5. Treat `info` findings as hygiene, not release blockers.

## Health score

`summary.health` summarizes the current visible findings. Do not optimize the score directly. Fix evidence-backed findings.

Suppressed findings do not reduce the score, but `summary.health.suppressedCount` shows accepted debt.

## Baselines and suppressions

Use a baseline when adopting Dr. Context in an existing repo:

```bash
drctx baseline --root . --output .drctx-baseline.json
cat > .drctx.json <<'JSON'
{
  "baseline": ".drctx-baseline.json"
}
JSON
drctx check --root . --config .drctx.json
```

Use `--show-suppressed` during audits:

```bash
drctx check --root . --config .drctx.json --show-suppressed
```

Do not use baselines to hide new context rot. Regenerate a baseline only after reviewing accepted findings.
```

# GitHub Action setup

Dr. Context can run in GitHub Actions and upload SARIF to code scanning.

## Minimal check

```yaml
name: Dr. Context

on:
  pull_request:
  push:
    branches: [main]

jobs:
  dr-context:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: CaptainEv1dence/dr-context@v0.3.9
        with:
          root: .
```

## SARIF upload

```yaml
permissions:
  contents: read
  security-events: write

jobs:
  dr-context:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: CaptainEv1dence/dr-context@v0.3.9
        with:
          root: .
          upload-sarif: 'true'
```

## Runtime constraints

The scanner itself is local and deterministic. The current composite Action may contact npm to install the selected `dr-context` package version before scanning. The scan does not call LLMs or external services.

Pin a release tag for repeatability. Use `version` only when you intentionally want the Action wrapper to run a different published npm package version.

Replace `v0.3.9` with the release tag you are validating when preparing a future release.
```

- [ ] **Step 1: Create coverage docs**

```md
# Recognized instruction surfaces

Dr. Context reports recognized local instruction surfaces. This is not a live claim about what any provider loaded at runtime.

Use the manifest to inspect recognized context:

```bash
drctx manifest --json --root .
drctx manifest --path src/example.ts --root .
```

## Tool families

| Tool family | Common local files Dr. Context recognizes |
| --- | --- |
| Vendor-neutral / Codex-style | `AGENTS.md`, nested `AGENTS.md` where applicable |
| Claude-style | `CLAUDE.md`, Claude Code Action workflow prompt inputs |
| GitHub Copilot | `.github/copilot-instructions.md`, `.github/instructions/*.instructions.md` |
| Cursor | `.cursor/rules/**/*.mdc`, scoped rule metadata |
| Gemini-style | `GEMINI.md` |
| Explicit guides | exact `AGENT_GUIDE.md` files |

## Reading coverage safely

- `agentInstructionFiles` are repo-visible instruction files.
- `workflowPrompts` are embedded workflow prompts and are not effective instruction files.
- `effectiveInstructionFiles` is path-scoped when `--path` is used.
- Parent workspace instructions are not inherited unless `--inherit-parent-instructions` is explicitly used in workspace scans.

## Known limitation

Dr. Context does not inspect a live Claude, Copilot, Cursor, Codex, or OpenCode session in 0.3.9. Agent probes are a later optional layer.
```
