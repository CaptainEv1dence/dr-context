# 0.3.8 Context Health Gate Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Decide whether context health scoring is stable enough to ship before 0.4, then implement only if the evidence says yes.

**Architecture:** Treat score as a deterministic summary over raw findings, never as a replacement for findings. If the scoring formula is not defensible after 0.3.5 through 0.3.7 dogfood, defer this plan to 0.4+.

**Tech Stack:** TypeScript, existing summary/reporting pipeline, docs.

## Candidate tasks

### Task 1: Stability review

- [ ] Review synthetic corpus results from 0.3.7.
- [ ] Review sanitized aggregate dogfood learnings if available.
- [ ] Count noisy finding families and decide whether scoring would mislead users.
- [ ] Commit the decision to this plan.

### Task 2: Score formula design, only if approved

- [ ] Define deterministic score inputs.
- [ ] Define how suppressed findings affect score.
- [ ] Define JSON summary fields and text rendering.
- [ ] Add tests for score calculation.
- [ ] Commit.

### Task 3: Badge or trend output, only if approved

- [ ] Decide whether to ship a badge recipe, a JSON field only, or no badge.
- [ ] Add docs and tests for selected output.
- [ ] Commit.

### Task 4: Release prep

- [ ] Update README, CHANGELOG, ARCHITECTURE, roadmap, and release docs.
- [ ] Run full verification gate from the umbrella spec.
- [ ] Run independent review.

---

## Hard deferral criteria

Defer this plan to 0.4+ if any of these are true:

- 0.3.5 or 0.3.6 findings still show meaningful false-positive risk.
- Score formula hides important raw finding details.
- Badge would encourage users to optimize for score rather than fixing context rot.
