# drctx init Design

## Goal

Help a repository adopt Dr. Context by generating starter config and instruction templates without overwriting user files.

## Scope

- Add a future `drctx init` command.
- Default to dry-run preview.
- Write files only with explicit `--write`.
- Never overwrite existing files unless a future `--force` is separately designed and confirmed.

## Candidate outputs

- `.drctx.json` with empty include/exclude and commented documentation omitted because JSON does not support comments.
- `AGENTS.md` starter only if no recognized agent instruction file exists.
- Optional `.github/workflows/dr-context.yml` only if explicitly requested by a flag in a later design.

## Safety rules

- No writes by default.
- No overwrite by default.
- Print every planned file path before writing.
- Keep templates synthetic and generic.
- Do not inspect or transmit private code beyond local file existence checks.

## Tests required before implementation

- Dry-run prints planned files and writes nothing.
- Existing `.drctx.json` is not overwritten.
- Existing `AGENTS.md` is not overwritten.
- `--write` creates only missing files.
- Windows and POSIX path output remains root-relative.

## Recommendation

Implement after `drctx explain` and docs validation. It is the safest first write-mode feature because it can be dry-run-first and no-overwrite by design.
```

# Safe Fixes Design

## Goal

Explore optional deterministic fixes for narrow context-rot findings while preserving user control.

## Scope

- Future `drctx fix` or `drctx check --fix-dry-run` command.
- Dry-run patch output before any write mode.
- Only deterministic, single-file, low-risk fixes.

## Explicitly unsafe for first version

- Rewriting broad instruction files.
- Guessing package-manager migration intent.
- Editing CI workflows with complex matrices.
- AI-generated remediation.

## Candidate first fixes

- Replace stale package script names when package.json has an exact current script and finding evidence names the stale script.
- Add a generated-file boundary sentence only when package metadata clearly names generated directories and an instruction file already exists.

## Safety rules

- Print unified diff by default.
- No writes without explicit `--write`.
- No multi-file fixes in first version.
- No fix if evidence has dynamic or ambiguous source.

## Recommendation

Do not implement before `drctx init`. Safe fixes are valuable but should come after one successful dry-run-first write-mode command.
```

# MCP and Agent Probe Boundary Design

## Goal

Define optional runtime-adjacent layers without weakening Dr. Context's deterministic scanner contract.

## MCP context gate

An MCP server can expose Dr. Context scan and manifest results to compatible agents. It should not change scan semantics.

Required boundaries:

- Local-only by default.
- No network calls except MCP protocol transport chosen by the user.
- Tools expose `check`, `manifest`, and `explain` results.
- Findings remain deterministic and source-backed.

## Agent probes

Agent probes may compare Dr. Context's expected context with what a live provider/editor appears to load, but this is not deterministic scanner truth.

Required labels:

- `experimental`.
- `provider-specific`.
- `not used for default findings`.
- `requires user opt-in`.

## Recommendation

Build MCP before probes. MCP reuses deterministic outputs. Probes create provider-specific support and trust complexity.
```

# Ruler and Repomix Integrations Design

## Goal

Explore integrations with tools that generate, route, or package agent context.

## Ruler integration

Potential value: Dr. Context can validate generated or synchronized agent instruction files after Ruler writes them.

Boundary: Dr. Context should not become a Ruler replacement or template engine.

## Repomix integration

Potential value: Dr. Context can scan repo context before a repository is packed for agent consumption.

Boundary: Dr. Context should not become a repo packer.

## First integration shape

- Documentation-first recipes.
- Optional CI examples.
- No runtime dependency on either tool.

## Recommendation

Start with docs recipes after `drctx explain` and `drctx init`. Add code integration only after users ask for a concrete workflow.
```

## Recommended order

1. `drctx explain <finding-id>` because it is local, read-only, and improves trust.
2. `drctx init` because it is the safest dry-run-first write-mode candidate.
3. Docs recipes for Ruler and Repomix because they test demand without dependencies.

5. Safe fixes because write-mode patches need stronger safety confidence.
6. Agent probes last because provider runtime claims are hardest to verify and support.

## Rule

Each item needs its own implementation plan before production code changes.
